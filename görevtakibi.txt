  priority                    TEXT DEFAULT 'normal',

  start_at                    TIMESTAMPTZ,
  has_specific_time           BOOLEAN DEFAULT false NOT NULL,
  timezone                    TEXT DEFAULT 'Europe/Istanbul' NOT NULL,
  is_recurring                BOOLEAN DEFAULT false NOT NULL,
  recurrence_rule_id          UUID,
  delegation_allowed          BOOLEAN DEFAULT false NOT NULL,
  approval_required           BOOLEAN DEFAULT false NOT NULL,
  closure_summary_required    BOOLEAN DEFAULT false NOT NULL,
  closure_file_required       BOOLEAN DEFAULT false NOT NULL,
  closure_image_required      BOOLEAN DEFAULT false NOT NULL,
  edit_due_date_allowed       BOOLEAN DEFAULT false NOT NULL,
  edit_schedule_allowed       BOOLEAN DEFAULT false NOT NULL,

─────────────────────────────────────────────────────────────────────────────
DURUM AKIŞI
─────────────────────────────────────────────────────────────────────────────

Durum (status) değerleri ve geçişleri:

  draft
    ↓ (yayınla)
  open
    ↓ (Göreve Başla — atanan)
  in_progress
    ↓ (Tamamla butonu)
  pending_completion_approval  ← approval_required = true ise
    ↓ (oluşturucu onaylar)
  completed

  open / in_progress  →  overdue          (vade geçti, terminal değil)
  overdue             →  not_completed    (incomplete_if_late = true ve vade geçtiyse)
  open                →  pending_approval  (upward_assignment talebi)
  open / in_progress  →  rejected         (geri gönderme)
  open / in_progress  →  soft_deleted     (görevi veren soft delete eder)
  soft_deleted        →  open             (restore)
  open / in_progress  →  cancelled        (oluşturucu iptal eder)

overdue terminal durum değildir; sadece gecikme göstergesidir.
not_completed ayrı bir performans statüsüdür.

─────────────────────────────────────────────────────────────────────────────
HİYERARŞİ VE ATAMA KURALLARI
─────────────────────────────────────────────────────────────────────────────

Görevler company_nodes tablosundaki pozisyon hiyerarşisine göre atanır.
Hiyerarşi parent_id alanı üzerinden kurulur.

Atama kuralları:
- Aynı seviyeye, aşağı seviyeye, kendine atanabilir
- Alt seviyeden üst seviyeye gönderim talep mantığındadır:
  "Bu görev hiyerarşi kuralı dışında; devam etmek istiyor musunuz?"
  onayı alındıktan sonra upward_assignment türünde approval_request oluşur
- Bir göreve birden fazla kişi atanabilir (task_participants, assignee türü)
- Bir görev birden fazla gözlemciye sahip olabilir (task_participants, watcher türü)
- Kullanıcı kendine görev atayabilir (kişisel todo kullanımı)

─────────────────────────────────────────────────────────────────────────────
DELEGE ETME KURALLARI
─────────────────────────────────────────────────────────────────────────────

delegation_allowed = true ise görevi kabul eden kişi devredebilir.

Delege akışı:
1. Delege eden → task_approval_requests'e 'delegation' kaydı açar
2. Delege alacak kişi kabul / ret edebilir
3. Ret durumunda orijinal atanan asli görevli kalır
4. Kabul durumunda delege alan kişi yeni assignee olur; orijinal assignee watcher'a düşer
5. Delege akışı sistem notu olarak chat'e ve task_history'ye işlenir

─────────────────────────────────────────────────────────────────────────────
GERİ GÖNDERME / RED KURALLARI
─────────────────────────────────────────────────────────────────────────────

Paralel veya alt seviyeden gelen görevler geri gönderilebilir.
Geri gönderme işlemi açıklama (reason) zorunluluğuyla yapılır.
Reason boşsa işlem tamamlanmaz.

Geri gönderilen görev:
- rejected statüsüne geçer
- Gerekçe task_approval_requests.reason alanında saklanır
- task_history'ye işlenir
- chat'e sistem notu olarak düşer
- Görev oluşturucuya bildirim gider

─────────────────────────────────────────────────────────────────────────────
SOFT DELETE / RESTORE KURALLARI
─────────────────────────────────────────────────────────────────────────────

Silme yetkisi yalnızca created_by_personnel_id sahibindedir.
Silme işlemi tasks.deleted_at alanına timestamp yazılarak yapılır.
tasks.status 'soft_deleted' olarak güncellenir.
Hard delete hiçbir koşulda uygulanmaz.

Soft deleted görevler:
- Normal listede gizlenir (is(deleted_at, null) filtresi)
- Yönetici/oluşturucu filtresiyle görülebilir
- Restore: deleted_at = null, status = önceki durum

─────────────────────────────────────────────────────────────────────────────
KAPATMA / ONAY KURALLARI
─────────────────────────────────────────────────────────────────────────────

closure_summary_required = true → tasks.closure_summary boş bırakılamaz
closure_file_required     = true → task_attachments'ta 'closure_file' kaydı olmalı
closure_image_required    = true → task_attachments'ta 'closure_image' kaydı olmalı
approval_required         = true → tamamlama isteği pending_completion_approval'a düşer;
                                    task_approval_requests'e 'closure_approval' kaydı açılır;
                                    oluşturucu onayladığında status = 'completed'

─────────────────────────────────────────────────────────────────────────────
TAMAMLANMADI / OVERDUE KURALLARI
─────────────────────────────────────────────────────────────────────────────

incomplete_if_late = true ise:
- Vade geçtiğinde atanan artık müdahale edemez
- status = 'not_completed'
- Oluşturucuya düşer; tarihleri güncelleyip tekrar gönderebilir
- Bu statü performans raporlarında kullanılır

Gecikme hesabı:
- has_specific_time = false → vade o günün 23:59'u (şube timezone'unda)
- Karşılaştırma: Date.now() vs due_at (timezone offset uygulanmış)
- Geciken görevler listede 'overdue' etiketi gösterir

─────────────────────────────────────────────────────────────────────────────
TARİH DEĞİŞTİRME KURALLARI
─────────────────────────────────────────────────────────────────────────────

edit_due_date_allowed = true → atanan sadece due_at değiştirebilir
edit_schedule_allowed = true → atanan start_at + due_at değiştirebilir
Tekrarlayan görevlerde tarih değiştirme yasaktır.

Tarih değişikliği:
1. tasks tablosu güncellenir
2. task_history'ye 'date_changed' kaydı açılır (metadata: eski ve yeni tarih)
3. Chat'e sistem notu düşer: "X kişisi tarihi Y olarak değiştirdi."

─────────────────────────────────────────────────────────────────────────────
TEKRARLAYAN GÖREVLER
─────────────────────────────────────────────────────────────────────────────

task_recurrence_rules tablosunda desteklenen frekanslar:
- daily    → her gün (belirli saatte)
- weekly   → belirli günlerde
- monthly  → ayın N'inci günü / ayın son günü / ayın N'inci haftanın X günü
- yearly   → belirli ay-gün / her N ayda bir
- interval → özel periyot (interval_value + birim)

src/lib/taskRecurrence.js adında yardımcı utility oluşturulmalıdır.
Bu utility mevcut proje tarzına uygun, bağımlılıksız (sadece native JS Date)

SAYFA YAPISI

Tasks.jsx iki ana moddan oluşur:
1. Liste görünümü (varsayılan)
2. Yeni Görev oluşturma formu (modal veya drawer olarak açılır)

Liste görünümü şu ana sekmelere ayrılır:
- Görevlerim       → participant_type = 'assignee' AND personnel_id = aktif personel
- Verdiğim Görevler→ created_by_personnel_id = aktif personel
- Gözlemci Olduklarım → participant_type = 'watcher' AND personnel_id = aktif personel

Her sekme altında durum filtresi:
- Devam Edenler  (status IN: open, in_progress, overdue, pending_approval,
                   pending_completion_approval)
- Tamamlananlar  (status = 'completed')

Ek filtreler (drawer veya panel):
- Geciken görevler   (status = 'overdue')
- Tamamlanmadı       (status = 'not_completed')
- Onay bekleyenler   (status IN: pending_approval, pending_completion_approval)
- Soft deleted       (deleted_at IS NOT NULL)
- Tarih aralığı
- Öncelik
- Kişi / pozisyon

Sıralama kriterleri:
- En yakın vade (varsayılan)
- En yeni oluşturulan
- En eski
- Öncelik (urgent → high → normal → low)
